## Current focus

**Step 2 — Unify the ML training and serving feature pipeline**

### Why this is next

Training and serving currently construct different ordered vectors. A model
associates meaning with each numeric position, so inference can return a result
while interpreting the wrong features.

### Goals

1. Define one named and ordered feature contract.
2. Reuse that contract for dataset generation, training, local prediction, and
   the HTTP inference service.
3. Retrain the model and prove parity with automated tests.

### Approach

1. Inventory the exact columns generated by `ml/dataset.py`, selected by
   `ml/train.py`, and built by both prediction paths.
2. Put feature names and extraction in one importable module.
3. Adjust the ML-service Docker build context if needed so it uses the shared
   module instead of maintaining a copy.
4. Train with deterministic settings and save the feature metadata with the
   model (or in a versioned model bundle).
5. Test that training and service extraction produce identical names, order,
   and values for the same board.
6. Retrain and copy the artifact to both runtime locations.

### Out of scope for Step 2

- Stopping the client timer / lobby rematch (Deferred)
- Environment variables, Redis, background timeouts
- Model-quality claims or large hyperparameter searches

### Success criteria

- Training and serving return the same feature vector for the same board.
- No scikit-learn feature-name warning during prediction.
- `make train`, `make test`, and the ML `/predict` smoke test pass.
